## JS 跑渲染停：那个困扰我很久的“草台班子”真相

前两天突然悟到一个前端最底层的逻辑，虽然入行这么多年，但那一刻才算真正把代码和浏览器运行联系在了一起：

**JS 跑，渲染就得停；渲染跑，JS 就得停。**

这两位就像是在抢一把椅子，谁坐上去了，另一位就得在门口站岗。如果 JS 霸占椅子时间太长，渲染进不来，用户看到的页面就会掉帧、卡顿，甚至死锁。

这种“互斥”关系，其实就是前端所有性能优化的逻辑原点。

### 1. 为什么不能“双赛道”？
很多人（包括以前的我）可能会想：现在电脑都是多核 CPU，为什么浏览器不能一边跑 JS 一边渲染页面？

原因很简单：**怕乱套**。
如果 JS 正在修改一个 DOM 节点的宽度，而渲染引擎同时在计算这个位置并绘图，那最后出来的效果到底是改之前的还是改之后的？为了保证页面表现的一致性，浏览器强制规定：**GUI 渲染线程与 JS 引擎线程互斥**。

### 2. 掉帧：就是 JS 抢戏太久
我们常说 60FPS 才算流畅，算下来一帧只有 **16.7ms**。
在这不到 17 毫秒的时间里，浏览器要干这么多事：
* 处理用户交互（点击、滚动）
* **执行 JS**
* 计算样式（Recalculate Style）
* 计算布局（Layout）
* 绘图（Paint）

如果你的 JS 逻辑（比如一个复杂的数组遍历或递归）跑了 100ms，那接下来的 5 帧渲染全部会被堵在门外。用户眼里看到的就是：点一下没反应，页面卡住了。

### 3. 我们这些年做的优化，到底在优什么？
明白了这个“抢椅子”的逻辑，回头看那些耳熟能详的技术点，你会发现它们其实都在给渲染“让路”：

* **React Fiber：** 为什么要搞时间切片？因为以前的同步渲染太霸道，椅子一坐到底。Fiber 把任务拆碎，每跑一小段就停下来问问浏览器：“渲染大叔，你要用椅子吗？”要用就先让你。
* **requestAnimationFrame (rAF)：** 以前用 `setTimeout` 做动画是盲目插队。rAF 则是“有序接力”，它会在浏览器下次重绘前精准触发 JS，保证动作和渲染节奏合拍。
* **Web Workers：** 既然主线程椅子少，那就另开一间房。把费劲的计算任务丢到 Worker 线程里去，算好了把结果传回来，别在主线程门口排队。
* **虚拟 DOM：** 别改一下 DOM 就让浏览器算一次布局。攒一波，一次性改完，让渲染线程只坐一次椅子，把活干完。

### 4. 感悟
以前觉得前端性能优化是背面试题，现在觉得更像是在做**主线程的时间管理**。

作为开发者，我们写的每一行代码其实都在和浏览器“分利”。如果 JS 拿得多了，给到渲染和交互的时间就少了。理解了这层互斥关系，写代码时自然就会多一个心眼：我这段逻辑，会不会让页面“断气”？

别觉得自己是“草台班子”，看透了这层底层博弈，你才算真正摸到了前端性能的门槛。
